# Style guide

## Language

Use **UK English** throughout:
- behaviour
- colour
- localisation
- organisation
- modelling
- centre
- analyse

## Code style

- Prefer explicit function names.
- Prefer small functions over monoliths.
- Favour docstrings for public interfaces.
- Avoid unexplained magic numbers.
- Keep plotting code centralised where practical.

## Notebook style

- Use clear markdown headings.
- Explain why the demo matters before showing outputs.
- End with “What we learned” and “Residual risks”.
- Avoid long hidden stateful cells.
